Lakebase is transitioning fully to Autoscaling.

* **[New instances are created as Autoscaling projects](#intro).** Since March 12, 2026, new Lakebase instances are created as Autoscaling projects, not Provisioned instances.
* **[Upgrade of existing Provisioned instances to Autoscaling](#provisioned-upgrade).** Upgrades begin in June 2026 for customers who have requested them, with remaining instance upgrades continuing in the following weeks.

## New instances are created as Autoscaling projects

When you create a new Lakebase instance, it is now created as a [Lakebase Autoscaling project](/aws/en/oltp/projects/) and appears on the **Autoscaling** page in the Lakebase app, not on the **Provisioned** page. Existing [Provisioned instances](/aws/en/oltp/instances/) are not affected and continue running exactly as before.

On the **Autoscaling** page, the new projects are marked with an information icon.

Hover over the icon to see a tooltip explaining that the project was created using the Database instance API, and that it supports both Database instance and Postgres APIs.

### What this means for you

* **Lakebase instance creation.** Any new Lakebase instance is created as a Lakebase Autoscaling project and appears on the **Autoscaling** page in the Lakebase app. You can manage these projects with both the [Database instance API](https://docs.databricks.com/api/workspace/database) and the [Postgres API](https://docs.databricks.com/api/workspace/postgres).
* **Create button in the UI.** The **Create** button (including in the Provisioned tab) now opens the Autoscaling project creation flow.
* **Autoscaling features.** New projects get autoscaling features, including autoscaling compute, scale-to-zero (enabled by default with a 24-hour inactivity timeout), branching, and instant restore. You can access these features from the Lakebase Autoscaling UI and [Postgres API](https://docs.databricks.com/api/workspace/postgres) when you're ready. For a full comparison with Provisioned, see the [Feature comparison](/aws/en/oltp/instances/#feature-comparison) table.
* **Existing Provisioned instances are unchanged.** They stay on the **Provisioned** page in the UI with the same behavior as before. Provisioned instances, connection strings, and APIs all continue working.
* **Your existing automation continues to work.** Existing automations keep working without modification, but new instances are created as Autoscaling projects.

### APIs

New instances created through the [Database instance API](https://docs.databricks.com/api/workspace/database) support both APIs, so existing automation continues without modification.

| Instance or project | API to use |
| --- | --- |
| New projects created via the Database instance API | Both APIs. You can manage the same project with either. |
| Projects created in the Autoscaling UI or with the Postgres API | [Postgres API](https://docs.databricks.com/api/workspace/postgres) only. |
| Existing Provisioned instances **before** upgrade to Autoscaling | [Database instance API](https://docs.databricks.com/api/workspace/database) only. |
| Existing Provisioned instances **after** upgrade to Autoscaling | Both APIs. You can manage the same instance with either. |

Existing DABs automation using `database_instances` continues to work. However, new instances created by `database_instances` are created as Autoscaling projects. For new Lakebase work, Databricks recommends using `postgres_projects` instead. See [bundle resources](/aws/en/dev-tools/bundles/resources#postgres_projects).

### Differences for new Lakebase instances

#### Compute size

Lakebase instances created as Autoscaling projects use Autoscaling **compute units (CUs)** instead of Lakebase Provisioned capacity units. When you create a project using the [Database instance API](https://docs.databricks.com/api/workspace/database), Lakebase maps your chosen Provisioned capacity unit to a min and max autoscaling CU range as follows:

note

**RAM per unit:** In Lakebase Provisioned, one capacity unit has 16 GB RAM. In Lakebase Autoscaling, one CU has 2 GB RAM.

| Provisioned capacity | Autoscaling min CU | Autoscaling max CU |
| --- | --- | --- |
| 1 (16 GB) | 4 (8 GB) | 8 (16 GB) |
| 2 (32 GB) | 8 (16 GB) | 16 (32 GB) |
| 4 (64 GB) | 16 (32 GB) | 32 (64 GB) |
| 8 (128 GB) | 48 (96 GB) | 64 (128 GB) |

For example, if your automation creates Lakebase instances with **capacity 1**, new Lakebase Autoscaling projects are created with **min CU 4** and **max CU 8** settings. You can change min/max later using the [Postgres API](https://docs.databricks.com/api/workspace/postgres) or in the Lakebase Autoscaling UI.

Pricing

All new instances, including those created using the [Database instance API](https://docs.databricks.com/api/workspace/database), run on the Autoscaling platform and use [Lakebase Autoscaling pricing](https://www.databricks.com/product/pricing/lakebase). With elastic compute replacing fixed-capacity billing, most customers see a reduction in compute costs. Existing Provisioned instances remain on their current pricing until upgraded. See [Upgrade of existing Provisioned instances to Autoscaling](#provisioned-upgrade).

#### Scale-to-zero

The read/write endpoint for new projects has **scale-to-zero enabled** by default with a **24-hour inactivity timeout**, so your compute automatically suspends after 24 hours of inactivity to reduce costs. You can adjust the timeout, or disable scale-to-zero to keep compute always running, in the Lakebase Autoscaling UI or using the [Postgres API](https://docs.databricks.com/api/workspace/postgres). See [Scale to zero](/aws/en/oltp/projects/scale-to-zero) for details.

#### History window

On **Lakebase Provisioned** (database instances), the **restore window** can be set up to **35 days**. On **Lakebase Autoscaling** (new projects), the **history window** setting has a maximum of **30 days**.

#### PostgreSQL version

New projects created through the [Database instance API](https://docs.databricks.com/api/workspace/database) or Lakehouse/Provisioned UI use **PostgreSQL 16**, aligned with Lakebase Provisioned. That means automation using the Database instance API creates projects with the same PostgreSQL version (16) as before. The default for projects created using the Autoscaling UI or [Postgres API](https://docs.databricks.com/api/workspace/postgres) is **PostgreSQL 17**.

#### Connection details and hostnames

Connection details for new projects use a different **hostname format** than Lakebase Provisioned. Provisioned instances use a **global** hostname (no region in the hostname). Newly created projects use a **regional** hostname (region included). Both read-write and read-only endpoints use the regional hostname format for new projects.

**Provisioned (global hostname):**

Text

```
host=instance-a1b2c3d4-e5f6-7890-abcd-ef1234567890.database.cloud.databricks.com
```

**Autoscaling (regional hostname):**

Text

```
host=ep-example-endpoint-a1b2c3d4.database.<region>.cloud.databricks.com
```

The regional hostname includes the region code for your cloud and region (for example, `us-east-1` on AWS or `eastus` on Azure).

important

If you use IP allowlists, you must allow the [regional ingress IPs](/aws/en/resources/ip-domain-region) for your region for new projects. See [Create IP access lists for workspaces](/aws/en/security/network/front-end/ip-access-list-workspace).

#### Private Link

Lakebase Autoscaling uses two Private Link endpoints: **front-end Private Link** for API access (workspace-level connectivity, unchanged) and **inbound Private Link for performance-intensive services** ([inbound Private Link for performance-intensive services](/aws/en/security/network/front-end/service-direct-privatelink)) for Postgres client connections. Whether you need inbound Private Link for performance-intensive services depends on how your applications connect.

| Your situation | What you need |
| --- | --- |
| You use Private Link, create a new instance, and connect from outside the Databricks workspace using a Postgres client | Add [inbound Private Link for performance-intensive services](/aws/en/security/network/front-end/service-direct-privatelink). |
| You already use Lakebase Autoscaling or other services that use inbound Private Link for performance-intensive services | No change. |
| You only use existing Provisioned instances and do not create new ones | No change. |

#### Permissions (ACLs)

Permissions on new Lakebase instances are stored on the [Lakebase project ACL](/aws/en/security/auth/access-control/#database-project) resource. You can manage them through the standard [Permissions API](https://docs.databricks.com/api/workspace/permissions) with `request_object_type=database-projects`, the Lakebase Autoscaling UI, or the [Postgres API](https://docs.databricks.com/api/workspace/postgres). Existing automation that uses the [Database instance API](https://docs.databricks.com/api/workspace/database) continues to work because those calls are proxied to the same `database-projects` resource. Reads and writes stay consistent across both API surfaces.

Earlier permissions model

Lakebase projects created with the [Database instance API](https://docs.databricks.com/api/workspace/database) or related tools (CLI, SDKs, Terraform, DABs) between March 12 and May 11, 2026 used an earlier permissions model where two independent ACL sets applied:

* **Database instance ACLs:** Set and evaluated through the [Database instance API](https://docs.databricks.com/api/workspace/database).
* **Lakebase project ACLs:** Set and evaluated through the Lakebase Autoscaling UI or the [Postgres API](https://docs.databricks.com/api/workspace/postgres).

The two sets can grant different levels of access. Review both [Database instance](/aws/en/security/auth/access-control/#database-instance) and [Lakebase project](/aws/en/security/auth/access-control/#database-project) ACL sets to understand the effective permissions. Databricks plans to unify permissions on these instances in a future update.

#### Project names and instance names

When you create a Lakebase instance through the [Database instance API](https://docs.databricks.com/api/workspace/database) or the Lakehouse/Provisioned UI, the instance name you specify becomes the project ID (`project_id`) of the resulting Lakebase Autoscaling project. The root branch is always created with the ID `production`.

Instance names and project IDs must be **DNS compliant**. Project ID uniqueness is **case insensitive**, which means you cannot have both `ABC` and `abc` as project IDs in the same workspace. When using the [Postgres API](https://docs.databricks.com/api/workspace/postgres), use the lowercase project ID. Creation fails if the project ID collides with an existing project ID.

Projects also have a display name (`display_name`), which is independent of the project ID and is not subject to the DNS compliance requirement.

#### Branch and project rename

Like instance names, project IDs and branch IDs cannot be changed after creation. However, projects also have a display name (`display_name`) that you can update at any time.

#### Child database instances

For **child** database instances (a child instance corresponds to a branch on the Autoscaling project):

* **Branch ID.** The child instance name you specify in the [Database instance API](https://docs.databricks.com/api/workspace/database) becomes the branch ID in the Autoscaling UI.
* **Branch limit.** With the [Postgres API](https://docs.databricks.com/api/workspace/postgres) or the Autoscaling UI, you can create any number of branches. With the [Database instance API](https://docs.databricks.com/api/workspace/database), you can create only one parent instance (one root branch) and one child instance (one child branch), which matches the previous Database instance API behavior.
* **Tags and budget.** Custom tags and usage policies are **inherited from the parent** branch. This is a change from the previous [Database instance API](https://docs.databricks.com/api/workspace/database) behavior, where child instances had their own separate values.
* **Deleting a child instance.** If you create additional branches using the Lakebase Autoscaling UI or Postgres API on a project that corresponds to a child instance, you cannot delete that child instance through the Lakehouse/Provisioned UI or API until all of those branches are deleted. Delete the branches first (using the Autoscaling UI or Postgres API), then delete the child instance.

#### Protected branches

If you protect a branch on a project created using the [Database instance API](https://docs.databricks.com/api/workspace/database) (whether the branch corresponds to a child instance or any other branch you created), you cannot delete that child instance or the root (parent) instance through the Lakehouse/Provisioned UI or API until the branch is unprotected. Unprotect the branch in the Lakebase Autoscaling UI first, then delete the instance. See [Protected branches](/aws/en/oltp/projects/protected-branches).

#### Private preview features

Newly created projects are Lakebase Autoscaling projects and do not support the Lakebase Provisioned **Data API** or **Forward ETL** private preview features. Those were separate offerings on Provisioned.

* **REST API access (PostgREST-style):** Lakebase Autoscaling has its own [Data API](/aws/en/oltp/projects/data-api) for PostgREST-compatible REST access (CRUD, query, RPC).

* **Syncing data to the Lakehouse:** Lakebase Autoscaling has [Lakebase Change Data Feed](/aws/en/oltp/projects/lakebase-cdf), which is different from the private preview Forward ETL on Lakebase Provisioned.
