Level of detail (LOD) expressions let you specify the granularity at which to calculate aggregations, independently of the fields (also called dimensions) in your query. This lets you compute metrics like percent of total, where the denominator aggregates at a coarser grain than the query.

## What are level of detail expressions?

Level of detail expressions enable you to specify exactly which fields to use when calculating an aggregate, regardless of the fields present in your query. This gives you fine-grained control over the scope of your calculations.

There are two types of level of detail expressions:

* **Fixed level of detail**: Aggregate over a pre-defined set of fields specified in the expression itself, ignoring other fields in the query.
* **Coarser level of detail**: Aggregate at a coarser granularity than the query by excluding specific fields from the grouping.

## Fixed level of detail

A fixed level of detail expression computes an aggregate at a granularity that you define, ignoring the fields in your query. In metric views, fixed LOD expressions are defined directly in the `expr` field of a column definition using SQL window functions with `PARTITION BY` clauses.

### When to use a fixed level of detail

Use fixed level of detail expressions when you need the following:

* **No dependency on query groupings**: Metrics with static partitioning across all uses.
* **Dataset-level aggregates**: Global aggregates compared with row-level groupings (for example, percent of total sales by priority).
* **Multi-level hierarchies**: Detail-level and rollup-level metrics available in the same metric view.

### Syntax

Fixed LOD expressions use [SQL window functions](/aws/en/sql/language-manual/sql-ref-window-functions) to compute aggregates at a defined granularity. Place the window function directly in the `expr` field of a field definition:

YAML

```
fields:  
  - name: <lod_name>  
    expr: <AGGREGATE_FUNCTION>(<column>) OVER (PARTITION BY <dim1>, <dim2>, ...)
```

To aggregate over the entire dataset, omit the `PARTITION BY` clause and leave empty parentheses after `OVER`.

### Example: total sales by order priority

Suppose you want to define a metric view where each order's sales can be compared alongside the total sales for its priority group. The following example computes `priority_total_price` in the source query and exposes it as an identity field:

YAML

```
version: 1.1  
  
source: samples.tpch.orders  
  
fields:  
  - name: order_priority  
    expr: o_orderpriority  
  - name: order_date  
    expr: o_orderdate  
  - name: priority_total_price  
    expr: SUM(o_totalprice) OVER (PARTITION BY o_orderpriority)  
  
measures:  
  - name: total_sales  
    expr: SUM(o_totalprice)  
  
  - name: pct_of_priority_total  
    expr: SUM(o_totalprice) / ANY_VALUE(priority_total_price)
```

The `priority_total_price` field defines the fixed total for each priority group directly in its `expr` field. The `pct_of_priority_total` measure divides individual order sales by that fixed total to produce a percentage, regardless of how the query groups results.

note

When referencing a fixed level of detail field in a measure expression, wrap it in an aggregate function. Use `ANY_VALUE` when the value is constant within a group, as in the previous example.

Create the metric view using SQL

To create this metric view outside Catalog Explorer, wrap the YAML in `CREATE OR REPLACE VIEW ... WITH METRICS LANGUAGE YAML AS` and place the definition between the `$$` delimiters:

SQL

```
CREATE OR REPLACE VIEW catalog.schema.sales_by_priority WITH METRICS LANGUAGE YAML AS  
$$  
  version: 1.1  
  
  source: samples.tpch.orders  
  
  fields:  
    - name: order_priority  
      expr: o_orderpriority  
    - name: order_date  
      expr: o_orderdate  
    - name: priority_total_price  
      expr: SUM(o_totalprice) OVER (PARTITION BY o_orderpriority)  
  
  measures:  
    - name: total_sales  
      expr: SUM(o_totalprice)  
  
    - name: pct_of_priority_total  
      expr: SUM(o_totalprice) / ANY_VALUE(priority_total_price)  
$$
```

The coarser level of detail example on this page follows the same pattern.

### Filtering on fixed level of detail expressions

Fixed level of detail expressions are computed before any query-time filters are applied. To apply a filter to a fixed LOD calculation, include the filter condition within the window function expression using a `CASE` statement or a `FILTER` clause.

## Coarser level of detail

A coarser level of detail expression aggregates at a coarser granularity than the query by excluding one or more fields from the partition. In metric views, coarser LOD expressions are implemented using [window measures](/aws/en/business-semantics/metric-views/advanced-techniques#window-measures) with the `all` range specification.

Experimental

Window measures are [Experimental](/aws/en/release-notes/release-types).

### When to use a coarser level of detail

Use coarser level of detail expressions when you need:

* **Dynamic groupings**: Aggregates that adapt to query groupings (for example, percent of total for any selected field).
* **Filter-aware aggregations**: Compute at a coarser granularity while respecting query-time filters.

### Syntax

For each field to exclude from the partition, define a window measure with `range: all`:

YAML

```
measures:  
  - name: <measure_name>  
    expr: <AGGREGATE_EXPRESSION>  
    window:  
      - order: <field_to_exclude>  
        range: all  
        semiadditive: last
```

To exclude multiple fields, add an entry to the `window` array for each field.

### Example: percent of total sales

To calculate the percentage of total sales for each order priority:

YAML

```
version: 1.1  
  
source: samples.tpch.orders  
  
fields:  
  - name: order_priority  
    expr: o_orderpriority  
  
measures:  
  - name: total_sales  
    expr: SUM(o_totalprice)  
  
  - name: all_priorities_sales  
    expr: SUM(o_totalprice)  
    window:  
      - order: order_priority  
        range: all  
        semiadditive: last  
  
  - name: pct_of_total_sales  
    expr: SUM(o_totalprice) / MEASURE(all_priorities_sales)
```

In this example:

* `total_sales` aggregates at the query's grouping level.
* `all_priorities_sales` uses `range: all` to compute a grand total across all order priorities, ignoring the `order_priority` field in the query.
* `pct_of_total_sales` divides priority-level sales by the grand total to produce a percentage.
