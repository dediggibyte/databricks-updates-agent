Classic compute resources in Databricks use flexible node types, which allows your compute resource to fall back to alternative, compatible instance types when your specified instance type is unavailable.

This behavior improves compute launch reliability by reducing capacity failures (stockout errors) during compute launches. For spot instances with fallback, flexible node types can attempt to acquire instances multiple times across different instance types before falling back to on-demand instances. This results in a higher percentage of instances running as spot instead of on-demand, reducing your total compute costs.

## How flexible node types works

When you launch a compute resource, your cloud provider sometimes runs out of capacity for your specified instance type. This results in a stockout error:

`AWS_INSUFFICIENT_INSTANCE_CAPACITY_FAILURE`

While these errors are more common for spot instances, they can occur for on-demand instances as well.

With flexible node types enabled, Databricks automatically generates or uses your specified fallback list of compatible instance types. If your preferred instance type is unavailable, Databricks attempts to acquire these backup instance types instead of failing immediately.

## Enable flexible node types in your workspace

Workspace admins can enable flexible node types in their workspace admin settings. When enabled, all new classic compute resources automatically fall back to alternate node types. Existing all-purpose compute resources are not affected. See [How are existing workloads affected?](#existing-workloads).

To enable flexible node types in a workspace:

1. As a workspace admin, go to the [settings page](/aws/en/admin/admin-concepts#admin-settings).
2. Click the **Compute** tab.
3. Toggle the **Enable auto flexible node types** setting:
   * **Enabled**: All new classic compute resources automatically use flexible node types unless explicitly disabled.
   * **Disabled**: Classic compute resources only use flexible node types if you explicitly configure `node_type_flexibility` in the compute resource configuration.

If this setting is disabled, users can still configure flexible node types for individual compute resources by explicitly configuring the `worker_node_type_flexibility` or `driver_node_type_flexibility` fields with custom fallback lists. To prevent users from configuring these fields, workspace admins can use compute policies. See [Flexible node type policy examples](/aws/en/admin/clusters/policy-definition#flexible-node-types).

## Specify a custom fallback list

When flexible node types are enabled in your workspace, Databricks automatically generates a fallback list of compatible instance types for new compute resources.

If you don't want to use the automatically generated fallback list, you can specify your own fallback list instead. Additionally, if flexible node types are disabled in your workspace, you can still specify a custom fallback list for your compute resource. Only certain instance types are compatible. See [Fallback instance type requirements](#requirements). For a reference of compatible instance types, see the [flexible node type compatibility reference](/aws/en/compute/flexible-node-type-instances).

Custom fallback lists are only supported when configuring compute using the API. See the [Clusters API](https://docs.databricks.com/api/workspace/clusters) reference documentation.

For example, the following configuration specifies which instance type the compute resource will fall back to if needed:

JSON

```
  "worker_node_type_flexibility": {  
    "alternate_node_type_ids": [  
      "Standard_E64as_v5"  
    ]  
  },  
  "driver_node_type_flexibility": {  
    "alternate_node_type_ids": [  
      "Standard_D64as_v5"  
    ]  
  },
```

### Fallback instance type requirements

Fallback instance types must be compatible with the compute's preferred instance type. Your list of fallback instance types must meet the following requirements:

* Same vCPU count and memory as the preferred instance type (fallback instances must have between 100% and 110% of the preferred instance type's memory)
* Same number of local disks and disk size as the preferred instance type
* Same CPU architecture as the preferred instance type (all ARM or all x86)
* Same OS image and Photon support as the preferred instance type
* No GPU instance types (GPUs are not supported)
* Maximum of 5 unique fallback instance types

* Virtual instance types (such as m-fleet) cannot be used as the preferred instance type or in the fallback list, as they already provide similar fallback behavior.

## Use flexible node types with pools

You can also customize a fallback list for pools. In the [Pools API](https://docs.databricks.com/api/workspace/instancepools), set the `node_type_flexibility` field to specify the fallback instance types. For example:

JSON

```
"node_type_flexibility": {  
    "alternate_node_type_ids": ["Standard_D64as_v5"]  
  }
```

Pools do not support using flexible instance types to maintain the minimum idle count. The pool can only launch VMs using the fallback instance types when a compute launch from the pool is attempted. Pre-warming the `minIdle` count only uses the preferred instance type.

## View the acquired instance types

When using flexible node types, your compute resource may consist of a mix of different instance types. All fallback instance types are compatible with your preferred type, maintaining the same vCPU count, memory, disk layout, CPU architecture, and OS image to ensure your workload runs correctly.

You can view which instance types were acquired for your compute resource:

1. In the compute details page, click the three dots next to the **Terminate** button and select **View JSON**.
2. Review the `node_type_id` field for each executor to see which instance types are running.

You can also use the [Get clusters info API](https://docs.databricks.com/api/workspace/clusters/get) to retrieve this information programmatically. Additionally, users with permission to access system tables can query the `node_timelines` table. See [Node timeline table schema](/aws/en/admin/system-tables/compute#node-timeline).

## Disable flexible node types on a compute resource

note

Databricks recommends keeping flexible node types enabled unless you have strict requirements for a specific instance type.

If you would prefer the compute launch to fail rather than fallback to an alternative instance type, you can disable the flexible node behavior at the individual compute-resource level. This is only supported when using the Clusters API. To disable flexible node types, set the flexible node type fields as empty in the compute configuration. For example:

JSON

```
"worker_node_type_flexibility": {  
  "alternate_node_type_ids": []  
},  
"driver_node_type_flexibility": {  
  "alternate_node_type_ids": []  
}
```

## Frequently asked questions

### How are existing workloads affected?

Existing all-purpose compute resources are unchanged. To use automatic fallback, create a new all-purpose compute resource after enabling the setting, or update the compute resource API spec with a custom fallback list.

For jobs using job compute, each run creates a new compute resource, so subsequent runs of existing jobs automatically use flexible fallback.

### Does this work with instance pools?

Yes. Flexible node types apply to your instance pool configurations. Some points to note:

* **Min idle stays consistent**: The pool's minimum idle instances (`minIdle`) are maintained using your primary node type. Any new VMs launched through a cluster launch request can be fulfilled using compatible fallback node types when the primary type is constrained.
* **Pool edits**: You cannot edit an instance pool after it's created. If you want to change any custom fallback settings, you must create a new instance pool.
* **API visibility**: The `/api/2.0/instance-pools/get` response does not show node type flexibility unless you have explicitly configured a custom fallback list on the pool. To see the fallback configuration of your instance pool, you can create a sample cluster and view the `/api/2.1/clusters/get` response.

### How does billing work?

You are billed based on the standard DBU rates for the instance types actually acquired. Any instance-level discounts you have with your cloud provider automatically apply to the matching instance types used by the compute resource.

### How does this interact with node type quotas in my workspace?

If a compute launch fails because your primary node type hits a "quota exceeded" limit, flexible node types can still improve launch reliability by automatically falling back to a compatible alternative. That said, for quota-driven failures, Databricks recommends treating fallback as a safety net rather than the primary fix. You can request a quota increase from your cloud provider so Databricks can consistently acquire your preferred instance types before resorting to alternatives, or use serverless compute.

### Can I enable flexible node types for only a subset of my workloads?

Auto-generated node type fallback is only configurable at the workspace level. However, you have two options to control the fallback behavior of a specific workload:

* **(Recommended)** Enable flexible node types for the workspace, then opt out a specific cluster by setting `alternate_node_type_ids` to an empty list `[]` in that cluster's spec.
* Disable flexible node types for the whole workspace, then opt in a specific cluster spec by providing a custom fallback list in `alternate_node_type_ids` that satisfies the [compatibility requirements](#requirements).
