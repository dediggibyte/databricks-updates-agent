**Applies to:**  Databricks SQL  Databricks Runtime

Inserts new rows into a table and optionally truncates the table or partitions. You specify the inserted rows by value expressions or the result of a query.

Databricks does not support `INSERT` for *Hive Avro* tables if the `timestamp-millis` type is present in the table schema.

## Syntax

```
[ common_table_expression ]  
  INSERT [ WITH SCHEMA EVOLUTION ] { OVERWRITE | INTO } [ TABLE ] table_name  
    [ PARTITION clause ]  
    [ ( column_name [, ...] ) | BY NAME ]  
    query  
  
[ common_table_expression ]  
  INSERT [ WITH SCHEMA EVOLUTION ] INTO [ TABLE ] table_name  
    [ BY NAME ]  
    [ REPLACE WHERE predicate | REPLACE USING ( column_name [, ...] ) ]  
    query  
  
[ common_table_expression ]  
  INSERT [ WITH SCHEMA EVOLUTION ] INTO [ TABLE ] table_name [target_alias]  
    [ BY NAME ]  
    REPLACE ON boolean_expression  
    { (query) [source_alias] | query }
```

note

When you `INSERT INTO` a Delta table, schema enforcement and evolution is supported.
If a column's data type cannot be safely cast to a Delta table's data type, a runtime exception is thrown.
If [schema evolution](/aws/en/tables/update-schema) is enabled, new columns can exist as the last columns of your schema (or nested columns) for the schema to evolve.

## Parameters

* **[common table expression](/aws/en/sql/language-manual/sql-ref-syntax-qry-select-cte)**

  Common table expressions (CTE) are one or more named queries which can be reused multiple times within the main
  query block to avoid repeated computations or to improve readability of complex, nested queries.
* **`WITH SCHEMA EVOLUTION`**

  **Applies to:**  Databricks Runtime 18.1 and above

  Enables [automatic schema evolution](/aws/en/tables/update-schema#merge-schema-evolution) for this `INSERT` operation.
  When enabled, the schema of the target Delta Lake table is automatically updated to accommodate additional columns or widened types of the source query. For a list of the supported type changes, see [Supported type changes](/aws/en/tables/features/type-widening#supported-type-changes).
* **`INTO` or `OVERWRITE`**

  If you specify `OVERWRITE` the following applies:

  + Without a `partition_spec` the table is truncated before inserting the first row.
  + Otherwise, all partitions matching the `partition_spec` are truncated before inserting the first row.

  If you specify `INTO` all rows inserted are additive to the existing rows.
* **[table\_name](/aws/en/sql/language-manual/sql-ref-names#table-name)**

  Identifies the table to be inserted to.
  The name must not include a [temporal specification](/aws/en/sql/language-manual/sql-ref-names#table-name).
  If the table cannot be found Databricks raises a [TABLE\_OR\_VIEW\_NOT\_FOUND](/aws/en/error-messages/table-or-view-not-found-error-class) error.

  `table_name` must not be a foreign table.

  **Applies to:**  Databricks Runtime 16.0 and above

  `table_name` may include an [options specification](/aws/en/sql/language-manual/sql-ref-names#table-name).
* **[PARTITION clause](/aws/en/sql/language-manual/sql-ref-partition#partition)**

  An optional parameter that specifies a target partition for the insert. You may also only partially specify the partition.

  When specifying a *static* partition (`column = value`), this column must not be repeated in the insert column list.

  When specifying a *dynamic* partition for `INSERT OVERWRITE`, the `table_name` must be a Delta Lake table.
* **( [column\_name](/aws/en/sql/language-manual/sql-ref-identifiers) [, …] )**

  An optional list of columns in the table. The insert command may specify any particular column from the table at most once.

  **Applies to:**  Databricks SQL  Databricks Runtime 12.2 LTS and above

  + If this command omits a column, Databricks SQL assigns the corresponding default value instead.
  + If the target table schema does not define any default value for the inserted column, Databricks SQL assigns `NULL` if the column is nullable. Otherwise, Databricks SQL raises an error.

  Providing no column list is equivalent to specifying all columns, except for those with assigned values in the `PARTITION` clause, in the order defined in the table.
* **`BY NAME`**

  **Applies to:**  Databricks SQL  Databricks Runtime 13.3 LTS and above

  When this clause is used instead of an explicit column list, the command uses the exposed column names of `query` to produce the column list in the order of `query`.
  As with an explicit column list, each column must exist in the target table, and must not be duplicated.
  If a column present in `table_name` is not part of the implied column list the `DEFAULT` value is used instead.

  `BY NAME` matches top-level columns by name.
  For nested struct fields, resolution is by name on Parquet (and other non-Delta) tables and by position on Delta Lake Lake tables.
  For Delta Lake tables, ensure the query struct field order matches the target. For example, use `named_struct('a', 1, 'b', 2)` for a column `STRUCT<a INT, b INT>`.

  No column in `query` may match a column specified in the `PARTITION` clause or any column that is generated.
* **REPLACE WHERE [boolean\_expression](/aws/en/sql/language-manual/sql-ref-expression)**

  **Applies to:**  Databricks SQL  Databricks Runtime 12.2 LTS and above  Delta Lake only

  Delete rows matching `boolean_expression` before inserting any rows matching `boolean-expression` specified in `query`. Rows in `query` that do not match `boolean_expression` are ignored.

  `table_name` must identify a Delta Lake table

  [boolean\_expression](/aws/en/sql/language-manual/sql-ref-expression) can only reference attributes from the table.

  [boolean\_expression](/aws/en/sql/language-manual/sql-ref-expression) is an expression that must return a `BOOLEAN`. The expression may contain the following operators, values, and functions:

  + `AND`, `OR`, and `NOT`
  + `IN` with literal values
  + `=`, `<=>`, `>`, `<`, `>=`, and `<=`
  + `TRUE` and `FALSE`
  + Functions `startsWith`, `endsWith`, `contains`
  + `IS NULL` and `IS NOT NULL`

  If the `boolean_expression` is not valid, Databricks raises a [DELTA\_REPLACE\_WHERE\_MISMATCH](/aws/en/error-messages/error-classes#delta_replace_where_mismatch) error.

  See [`REPLACE WHERE`](/aws/en/delta/selective-overwrite#replace-where).
* **REPLACE USING ( [column\_name](/aws/en/sql/language-manual/sql-ref-identifiers) [, …] )**

  **Applies to:**  Databricks SQL  Databricks Runtime 16.3 and above

  Atomically deletes rows from `table_name` that match a row in `query` on the specified columns, then inserts the rows from `query`. Matching is based on equality comparison of the specified column values in the table and `query`.

  + For Databricks Runtime 16.3 to 17.1, this feature requires a partitioned table. The specified columns in the `USING` clause must include all of the table's partition columns.
  + For Databricks Runtime 17.2 and above, partitioned tables, unpartitioned tables, and tables with liquid clustering are supported. The specified columns in the `USING` clause can include any combination of the table's columns.

  `BY NAME` support for `REPLACE USING` applies to Databricks Runtime 18.1 and above.

  note

  Similar to `JOIN USING`, `REPLACE USING` matches rows using regular equality comparison, where `NULL` values are treated as not equal to anything. Rows with `NULL` values in the specified columns won't match and therefore aren't removed from the target table.
* **[query](/aws/en/sql/language-manual/sql-ref-syntax-qry-query)**

  A query that produces the rows to be inserted.

  You must match the number of columns returned by the query with the specified or implied insert column list.

  If a data type cannot be safely cast to the matching column data type, a runtime exception is thrown.

  **Applies to:**  Databricks SQL  Databricks Runtime 11.3 LTS and above

  + If `query` consists of a [VALUES](/aws/en/sql/language-manual/sql-ref-syntax-qry-select-values) clause the `expression` can be `DEFAULT`.
  + If `query` consists of a [SELECT](/aws/en/sql/language-manual/sql-ref-syntax-qry-select) clause the `named_expression` can be `DEFAULT`.
  + `DEFAULT` will insert the explicitly defined `DEFAULT` expression of the corresponding column in `table_name`, or `NULL` if none is defined.

  If [schema evolution](/aws/en/tables/update-schema) is enabled, new columns can exist as the last columns of your schema (or nested columns) for the schema to evolve.
* **[target\_alias](/aws/en/sql/language-manual/sql-ref-names#table-alias)**

  An optional [table alias](/aws/en/sql/language-manual/sql-ref-names#table-alias) for the table for `REPLACE ON`. The alias must not include a column list.
* **[source\_alias](/aws/en/sql/language-manual/sql-ref-names#table-alias)**

  An optional [query alias](/aws/en/sql/language-manual/sql-ref-names#table-alias) for the `query` for `REPLACE ON`. The alias must not include a column list.
* **REPLACE ON [boolean\_expression](/aws/en/sql/language-manual/sql-ref-expression)**

  **Applies to:**  Databricks SQL  Databricks Runtime 17.1 and above

  If `table_name` is a Delta Lake table, `REPLACE ON` atomically deletes any rows in the table that match the `query` rows based on the specified [boolean\_expression](/aws/en/sql/language-manual/sql-ref-expression), then inserts the `query` rows. Unlike `REPLACE WHERE`’s [boolean\_expression](/aws/en/sql/language-manual/sql-ref-expression), `REPLACE ON`’s [boolean\_expression](/aws/en/sql/language-manual/sql-ref-expression) can reference columns from both the table and the `query`. If the `query` is empty, no deletions occur because there are no `query` rows to match.

  `BY NAME` support for `REPLACE ON` applies to Databricks Runtime 18.1 and above.

### INSERT INTO

#### INSERT using VALUES

SQL

```
> CREATE TABLE students (name VARCHAR(64), address VARCHAR(64) DEFAULT 'unknown', student_id INT)  
  PARTITIONED BY (student_id);  
  
-- Single row insert using a `VALUES` clause specifying all columns.  
> INSERT INTO students VALUES  
    ('Amy Smith', '123 Park Ave, San Jose', 111111);  
  
-- Single row insert using an implicit default for address  
> INSERT INTO students(name, student_id) VALUES('Grayson Miller', 222222);  
  
-- Single row insert using an explicit DEFAULT keyword for address  
> INSERT INTO students VALUES('Youna Kim', DEFAULT, 333333);  
  
--  Multi-row insert using a `VALUES` clause  
> INSERT INTO students VALUES  
    ('Bob Brown', '456 Taylor St, Cupertino', 444444),  
    ('Cathy Johnson', '789 Race Ave, Palo Alto', 555555);  
  
-- Multi-row insert using a mix of DEFAULT and literals  
> INSERT INTO students VALUES  
    ('Gwyneth Zhao', '120 Main St, Rockport', 666666),  
    ('Jackson Peterson', DEFAULT, 777777);  
  
> SELECT * FROM students;
