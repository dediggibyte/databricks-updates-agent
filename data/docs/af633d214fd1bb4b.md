Databricks SQL alerts run queries on a schedule and notify you when a condition that you define is met against the query result. When you schedule an alert, its associated query runs and the condition is evaluated. You can also view an alert history to review the results of past evaluations.

To learn how to work with legacy alerts instead, see [What are legacy alerts?](/aws/en/sql/user/alerts/legacy).

## What you can do with alerts

Alerts let you monitor any SQL query result on a schedule. Use them to track business KPIs, monitor data quality, watch cost trends, and catch operational issues across your Databricks workloads. Common patterns across Databricks include:

* **Monitor business metrics in metric views**: Reference a Unity Catalog metric view by its fully qualified name in the alert query to monitor governed business metrics. See [Alert on metric views](/aws/en/sql/user/alerts/query-patterns#metric-views).
* **Detect data quality issues and anomalies**: Pair alerts with Unity Catalog data quality monitors and anomaly detection so an unexpected metric, distribution shift, or profile change sends a notification. See [Alerts for anomaly detection](/aws/en/data-governance/unity-catalog/data-quality-monitoring/anomaly-detection/alerts) and [Profile alerts](/aws/en/data-governance/unity-catalog/data-quality-monitoring/data-profiling/monitor-alerts).
* **Track usage and cost**: Build alerts on system tables for serverless billing or ingestion to catch unexpected spend. See [Monitor the cost of serverless compute](/aws/en/admin/system-tables/serverless-billing) and [Monitor managed ingestion pipeline cost](/aws/en/ingestion/lakeflow-connect/monitor-costs).
* **Watch SQL warehouse and query health**: Alert on warehouse events or query history to catch slow queries, failed sessions, or capacity issues. See [Example queries for monitoring SQL warehouse activity](/aws/en/compute/sql-warehouse/monitor/queries) and [Warehouse events system table reference](/aws/en/admin/system-tables/warehouse-events).
* **Audit access and security events**: Alert on audit log queries to flag unusual workspace activity. See [Monitor Genie Agents usage with audit logs and alerts](/aws/en/ai-bi/admin/audit).
* **Catch failures in AI agents**: Alert on agent quality metrics so failures and emerging issues surface during development and operation. See [Agent development lifecycle](/aws/en/agents/agents-dev-lifecycle).
* **Run an alert as a task in Lakeflow Jobs**: Add an alert as a task so condition checks run on a pipeline trigger and downstream tasks can branch on the result. See [SQL alert task for jobs](/aws/en/jobs/tasks/alert).

## Differences from legacy alerts

The latest version of Databricks SQL alerts behaves differently from legacy alerts in a few key ways:

* **Query reuse**: An existing saved SQL query cannot be reused when creating an alert. Each alert owns its query definition, which can be authored directly in the new alert editor.
* **Alert status values**: Alert states are simplified and alerts no longer support the `UNKNOWN` status from legacy alerts. Evaluations resolve to `OK`, `TRIGGERED`, or `ERROR`.

You can continue to use both the latest alerts and legacy alerts side by side while you transition.
