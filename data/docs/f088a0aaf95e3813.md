After you create and develop your Databricks app, deploy it to make it accessible in the Databricks workspace. Deployment builds your app, installs dependencies, and runs it using the configuration defined in your project files. You can deploy apps using the Databricks UI or the Databricks CLI.

note

If you create an app from a template, Databricks deploys it automatically when you first create it. However, you can still re-deploy it later after you make changes. See [Create a Databricks app from a template](/aws/en/dev-tools/databricks-apps/create-app-template).

## Deployment logic

Databricks Apps supports deploying applications that use either Python, Node.js, or a combination of both. This allows for flexible architectures, such as a Node.js frontend with a Python backend.

During deployment, the build process checks for a `package.json` file at the root of your app directory to determine whether Node.js is used. If present, it includes Node-specific build steps alongside Python steps. The deployment logic follows this pattern:

**If `package.json` is present**:

1. Install Node.js dependencies. If `pnpm-lock.yaml` is present, run `pnpm install --frozen-lockfile`; otherwise, run `npm install`. See [Define Node.js dependencies](/aws/en/dev-tools/databricks-apps/dependencies#node-dependencies).
2. Install Python dependencies:
   * If `requirements.txt` exists, run `pip install -r requirements.txt`
   * If `requirements.txt` doesn't exist and both `pyproject.toml` and `uv.lock` exist, run `uv sync` to install dependencies
3. Run the `build` script if one is defined in `package.json` (`npm run build` or `pnpm run build`).
4. Run the command specified in `app.yaml`. For `npm` apps, if no command is specified, Databricks runs `npm run start`. `pnpm` apps must specify a command in `app.yaml`.

note

For `npm` apps, if no command is specified in `app.yaml`, Databricks executes `npm run start`, even if the app includes Python code. To run both Python and Node.js processes, define a custom `start` script that uses a tool like [`concurrently`](https://www.npmjs.com/package/concurrently) to launch both. For example: `concurrently "npm run start:node" "python my_app.py"`.

**If `package.json` is not present**:

1. Install Python dependencies:
   * If `requirements.txt` exists, run `pip install -r requirements.txt`
   * If `requirements.txt` doesn't exist and both `pyproject.toml` and `uv.lock` exist, run `uv sync` to install dependencies
2. Run the command specified in `app.yaml`, or `python <my-app>.py` if no command is specified

## Prepare for deployment

Before you deploy your app, verify that your project includes the necessary components:

* **Main script** - Your entry point file, such as `app.py` or `app.js`.
* **Optional `app.yaml` file** - If your app requires a custom command or environment variables, include an `app.yaml` file to configure execution. See [Configure Databricks app execution with `app.yaml`](/aws/en/dev-tools/databricks-apps/app-runtime).
* **Dependencies** - Make sure all dependencies are available. See [Manage dependencies for a Databricks app](/aws/en/dev-tools/databricks-apps/dependencies).
* **Secrets or environment values** - If you use the `env` section in `app.yaml`, verify that the referenced secrets or external sources are correctly configured and accessible. See [Add resources to a Databricks app](/aws/en/dev-tools/databricks-apps/resources).

In addition, make sure the app [service principal](/aws/en/admin/users-groups/service-principals) has access to the source code folder.

## Choose a deployment source

You can deploy Databricks apps from the following sources:

* **Workspace folder** - Upload app files to a workspace folder and deploy from there. This is the standard deployment method. See [Deploy from a workspace folder](#deploy-workspace).
* **Git repository** - Configure a Git repository for an app and deploy directly without uploading files to the workspace. The app reads code from the configured Git reference (branch, tag, or commit) each time you deploy. See [Deploy from a Git repository](#deploy-git).

You can switch between workspace and Git sources for the same app at any time. See [Update or redeploy the app](#update-redeploy).

## Deploy the app

The following sections describe how to deploy from a workspace folder or directly from a Git repository.

### Deploy from a workspace folder

* Databricks UI
* Databricks CLI

To deploy an app from the Databricks UI:

1. Upload the app files to your Databricks workspace. For instructions, see [Import a file](/aws/en/files/workspace-basics#import-a-file).
2. In your Databricks workspace, click the  app switcher and select **Databricks Apps**.
3. Select your app in the **Name** column.
4. Click **Deploy** and select the folder in your workspace where you uploaded the app files.
5. Click **Select**, then **Deploy**.

To deploy an app using the CLI:

1. Open a terminal and navigate to the directory that contains your app files.
2. Upload your app files to the Databricks workspace using the `sync` command. Replace the path with the workspace location where you want to upload the files.

   Bash

   ```
   databricks sync --watch . /Workspace/Users/my-email@org.com/my-app
   ```

   The `--watch` flag keeps the sync process running and automatically uploads changes when you modify files locally. To exclude specific files or directories from syncing, add them to a `.gitignore` file in your local app directory. Common files to exclude are `node_modules/`, `.env`, `__pycache__/`, `.DS_Store`, and any large data files or build artifacts.
3. Verify the upload by viewing the files in your workspace. Click the  app switcher > **Analytics and AI**, then click  **Workspace** and navigate to the directory you created for your app.
4. Deploy the app by running the following command. Replace the app name and source code path with your values.

   Bash

   ```
   databricks apps deploy my-app-name \  
      --source-code-path /Workspace/Users/my-email@org.com/my-app
   ```

   The CLI displays deployment progress and confirms when the app is running.

### Deploy from a Git repository

To deploy an app from a Git repository, add the repository at the app level, then specify the Git reference when you deploy it. Your Git repository must contain the app files, including `app.yaml`, dependencies, and entry point. All major Git providers are supported, including GitHub, GitLab, and Bitbucket. You can also deploy Git-backed apps using the UI, CLI, API, or Declarative Automation Bundles.

* Databricks UI
* Databricks CLI

To configure and deploy an app from Git:

1. Upload your app files to a Git repository.
2. In your Databricks workspace, click the  app switcher and select **Databricks Apps**.
3. Either select an existing app to edit, or click **+ Create app** and then **Create a custom app**. See [Create a custom Databricks app](/aws/en/dev-tools/databricks-apps/create-custom-app).
4. In the **Configure Git** step, enter the Git repository URL (for example, `https://github.com/org/repo`) and select a Git provider.
5. For a new app, enter the **Git reference (branch, tag, or commit)** and, for GitHub repositories, optionally enable **Auto deploy on push events**. See [Enable automatic Git deployments](#automatic-deployments).
6. Click **Create app** or **Save** to return to the app overview page.
7. For private repositories, the app's service principal must have a Git credential configured. On the app overview page, click **Configure Git credential**. You must have `CAN MANAGE` permissions on the app to add a Git credential. Public repositories don't require a Git credential. For instructions for each provider, see [Connect your Git provider to Databricks](/aws/en/repos/get-access-tokens-from-git-provider).

Then, deploy the app:

1. On the app overview page, click **Deploy**.
2. Select **From Git**.
3. For **Git reference**, enter the branch name, tag, or commit SHA (for example, `main`, `v1.0.0`, or a commit hash).
4. For **Reference type**, specify the type of reference, such as a branch, tag, or commit.
5. (Optional) For **Source code path**, enter a path to a specific directory within the repository. The app treats that directory as the top-level directory and can't access files outside of it. If you don't specify a path, Databricks uses the repository root.
6. (Optional) To deploy automatically on every commit to the branch, enable **Auto deploy on push events**. You must set a **Branch to watch**. Optionally, you can include a **Source code path** to set a path other than the repo root. See [Enable automatic Git deployments](#automatic-deployments).
7. Click **Deploy**.

To deploy an app from Git using the CLI:

1. Upload your app files to a Git repository.
2. Configure the Git repository on the app when you create the app, or add it to an existing app. Supported providers include `gitHub`, `gitHubEnterprise`, `gitLab`, `gitLabEnterpriseEdition`, `bitbucketCloud`, `bitbucketServer`, `azureDevOpsServices`, and `awsCodeCommit`.

   To create a new app with a Git repository configured, use the [create](/aws/en/dev-tools/cli/reference/apps-commands#databricks-apps-create) command:

   Bash

   ```
   databricks apps create my-app \  
      --json '{"git_repository": {"url": "https://github.com/org/repo", "provider": "gitHub"}}'
   ```

   To add or update a Git repository on an existing app, use the [create-update](/aws/en/dev-tools/cli/reference/apps-commands#databricks-apps-create-update) command:

   Bash

   ```
   databricks apps create-update my-app \  
      --json '{"update_mask": "git_repository", "git_repository": {"url": "https://github.com/org/repo", "provider": "gitHub"}}'
   ```
3. For private repositories, configure a Git credential for the app's service principal. You must have `CAN MANAGE` permissions on the app to add a Git credential. Public repositories don't require a Git credential.

   To add a Git credential to the app's service principal using the CLI, run:

   Bash

   ```
   databricks git-credentials create --json '{  
     "git_provider": "gitHub",  
     "git_email": "your-email@example.com",  
     "personal_access_token": "YOUR_TOKEN",  
     "principal_id": YOUR_SP_ID,  
     "name": "GitHub credentials for SP"  
   }'
   ```

   Replace `YOUR_SP_ID` with the app's service principal ID. For instructions to get a personal access token for each provider, see [Connect your Git provider to Databricks](/aws/en/repos/get-access-tokens-from-git-provider).
4. Deploy the app by specifying the Git reference. You can specify a `branch`, `tag`, or `commit` (these are mutually exclusive). Optionally, include `source_code_path` to deploy from a subdirectory within the repository.

   Bash

   ```
   databricks apps deploy my-app \  
      --json '{"git_source": {"branch": "main"}}'
   ```

   To deploy a specific tag or commit:

   Bash

   ```
   databricks apps deploy my-app \  
      --json '{"git_source": {"tag": "v1.0.0"}}'
   ```

   Bash

   ```
   databricks apps deploy my-app \  
      --json '{"git_source": {"commit": "abc123def456"}}'
   ```

   To deploy from a subdirectory within the repository:

   Bash

   ```
   databricks apps deploy my-app \  
      --json '{"git_source": {"branch": "main", "source_code_path": "apps/my-app"}}'
   ```

   The CLI displays deployment progress and confirms when the app is running.

For branch or tag references, Databricks deploys the most recent commit from that branch or tag. For commit SHA references, Databricks always deploys that specific commit. If the service principal's Git credential is invalid or expired, the deployment fails.

note

Apps created before Git deployment became generally available don't automatically grant creators `CAN MANAGE` permissions on the app's service principal. If you need to add a Git credential to an older app, ask a workspace admin to grant you `CAN MANAGE` permissions on the service principal.

Service principals support one Git credential per provider. If you update the credential elsewhere, such as through the account console, it replaces the existing credential for that provider.

#### Enable automatic Git deployments

Beta

Automatic deployments from Git are in [Beta](/aws/en/release-notes/release-types). GitHub is the only supported provider during Beta.

When you enable automatic deployments, Databricks creates a webhook on your GitHub repository and redeploys your app each time the configured branch receives a new commit. Your deployed app stays in sync with your repository without any manual steps. Repositories are limited to 20 webhooks (250 for GitHub Enterprise).

Automatic deployments require the following setup:

* The Databricks GitHub app must be installed on the repository.
* The repository must be private. Databricks doesn't support automatic deployments from public repositories.
* The app's service principal must have a Git credential with access to the repository. See [Connect your Git provider to Databricks](/aws/en/repos/get-access-tokens-from-git-provider).

To enable automatic deployments, configure GitHub as the Git provider and enable **Auto deploy on push events**. You can do this in either of the following places:

* The **Configure Git** step of the Create custom app wizard. See [Create a custom Databricks app](/aws/en/dev-tools/databricks-apps/create-custom-app).
* The **Auto-deployment** badge in an existing app or by reconfiguring **App Settings**. Follow the steps in [Deploy from a Git repository](#deploy-git).

The Git reference must be a branch name. Tags aren't compatible with automatic deployments.

note

If the app's service principal doesn't have a [Git credential](/aws/en/repos/get-access-tokens-from-git-provider) for the repository, Databricks prompts you to add one when you enable automatic deployments. Authorizing access enables automatic deployments, but you must manually redeploy the app to start them.
