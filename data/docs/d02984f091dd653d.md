Add [Unity Catalog tables](/aws/en/tables/) as Databricks Apps resources so your app can query and modify data stored in Unity Catalog with governance and access control. Unity Catalog tables provide structured data storage with fine-grained permissions, so your app can securely read and write data without hardcoding credentials.

## Privilege requirements

To access a Unity Catalog table, the app's service principal must have the `USE CATALOG` privilege on the parent catalog, the `USE SCHEMA` privilege on the parent schema, and the `SELECT` or `MODIFY` privilege on the table. When you add the table resource, Databricks automatically grants these privileges to the app's service principal.

For this automatic granting to succeed, one of the following must be true for each privilege:

* **For `USE CATALOG`:** Either all account users have the `USE CATALOG` privilege on the catalog, or you have the `MANAGE` privilege on the catalog.
* **For `USE SCHEMA`:** Either all account users have the `USE SCHEMA` privilege on the schema, or you have the `MANAGE` privilege on the schema.
* **For `SELECT` or `MODIFY`:** Either all account users have the `SELECT` or `MODIFY` privilege on the table, or you have the `MANAGE` privilege on the table.

See [Unity Catalog privileges reference](/aws/en/data-governance/unity-catalog/access-control/privileges-reference).

## Add a Unity Catalog table resource

Before you add a table as a resource, review the [app resource prerequisites](/aws/en/dev-tools/databricks-apps/resources#resources-prereqs).

1. In the **App resources** section when you create or edit an app, click **+ Add resource** > **UC table**.
2. Choose a Unity Catalog table from the available tables in your workspace. The table must already exist in Unity Catalog.
3. Select the appropriate permission level for your app:
   * **Select:** Grants the app permission to query and read data from the table. This is appropriate for apps that only need to read data.
   * **Modify:** Grants the app permission to insert, update, and delete data in the table. This permission implicitly includes `SELECT`, so the app can also read data.
4. (Optional) Specify a custom resource key, which is how you reference the table in your app configuration. The default key is `table`.

note

`MODIFY` is not available for all table types. For example, views don't support the `MODIFY` privilege. If the table doesn't support `MODIFY`, only `SELECT` is available.

## Environment variables

When you deploy an app with a Unity Catalog table resource, Databricks exposes the table's full three-level name through environment variables that you can reference using the `valueFrom` field.

**Example configuration:**

YAML

```
env:  
  - name: UC_TABLE_NAME  
    valueFrom: table # Use your custom resource key if different
```

**Using the table in your application:**

Python

```
import os  
from databricks.sdk import WorkspaceClient  
  
# Access the table name  
table_name = os.getenv("UC_TABLE_NAME")  
  
# Initialize workspace client  
w = WorkspaceClient()  
  
# Query the table using a SQL warehouse resource  
result = w.statement_execution.execute_statement(  
    warehouse_id=os.getenv("DATABRICKS_WAREHOUSE_ID"),  # Requires a SQL warehouse resource  
    statement=f"SELECT * FROM {table_name} LIMIT 10"  
)
```

For more information, see [Use environment variables to access resources](/aws/en/dev-tools/databricks-apps/environment-variables#access-resources).

## Remove a Unity Catalog table resource

When you remove a Unity Catalog table resource from an app, the app's service principal loses the `SELECT`, `MODIFY`, and `ALL PRIVILEGES` grants on the table. Databricks also attempts to revoke hierarchical catalog and schema permissions. The table itself remains unchanged and continues to be available for other users and applications that have appropriate permissions.

## Best practices

Consider the following when you work with Unity Catalog table resources:

* Grant minimal permissions. Use `SELECT` unless your app needs to write data to the table.
* Combine table resources with a [SQL warehouse resource](/aws/en/dev-tools/databricks-apps/sql-warehouse) when your app needs to run SQL queries against the table.
* Use separate table resources for different tables rather than granting broad schema-level access, so that you can apply the principle of least privilege.
* Consider using [Unity Catalog volumes](/aws/en/dev-tools/databricks-apps/uc-volumes) for unstructured data and Unity Catalog tables for structured data.
