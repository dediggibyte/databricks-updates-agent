This page describes how Unity Catalog credential vending functionality supports access to data in Databricks from external processing engines.

Credential vending supports external systems that connect to Unity Catalog using the Unity REST API and Apache Iceberg REST catalog. See [Access Databricks tables from Delta clients](/aws/en/external-access/unity-rest) and [Access Databricks data using external systems](/aws/en/external-access/).

## What is Unity Catalog credential vending?

*Credential vending* grants short-lived credentials using the Unity Catalog REST API. The granted credentials inherit the privileges of the Databricks principal used to configure the integration. There are two types of credential vending:

* **Table credential vending** provides access to data registered in your Unity Catalog metastore.
* **Path credential vending** provides access to external locations in your Unity Catalog metastore.

## Requirements

* External access must be configured on the metastore with `EXTERNAL USE SCHEMA` granted to the requesting principal. See [Enable external data access on the metastore](/aws/en/external-access/admin#external-data-access).
* The workspace URL must be accessible to the requesting engine, including engines behind [IP access lists](/aws/en/security/network/front-end/ip-access-list-workspace) or [AWS Private Link](/aws/en/security/network/classic/privatelink).
* Cloud storage URLs must be accessible through firewall and network controls.

## Table credential vending

Table credentials include a short-lived access token string and cloud storage location URL that the external engine can use to access table data and metadata from the cloud storage location.

### Supported access types

Table credential vending supports the following table types and operations:

| Table type | Read | Write | Create |
| --- | --- | --- | --- |
| Managed Delta | Yes | Yes\* | Yes\* |
| External Delta | Yes | Yes | Yes |
| Managed Iceberg | Yes | Yes | Yes |
| Foreign Iceberg | Yes | No | No |
| Delta with Iceberg reads (UniForm) | Yes | Yes\*\* | No |

\* Creating and writing to Unity Catalog managed tables from Delta clients is in [Public Preview](/aws/en/release-notes/release-types).

\*\* After externally writing to a UniForm table from a Delta client, run `MSCK REPAIR TABLE` to generate Iceberg metadata.

note

Some clients support access to tables backed by Delta Lake, while others require that you enable Iceberg reads (UniForm) on tables. See [Read Delta Lake tables with Iceberg clients using UniForm](/aws/en/delta/uniform).

### Request a temporary table credential for external data access

Support for credential vending varies by external client. Where supported, the client should automatically leverage vended credentials when a connection is configured.

This section provides an example of explicitly calling the credential vending API endpoint. Some external clients might require you to explicitly set configurations to access data and metadata in cloud object storage backing your Unity Catalog tables. You can use values returned by credential vending to configure access.

note

You can retrieve a list of tables that support credential vending by invoking the ListTables API with the `include_manifest_capabilities` option enabled. Only tables marked `HAS_DIRECT_EXTERNAL_ENGINE_READ_SUPPORT` or `HAS_DIRECT_EXTERNAL_ENGINE_WRITE_SUPPORT` are eligible for reference in the temporary-table-credentials API. See [GET /api/2.1/unity-catalog/tables](https://docs.databricks.com/api/workspace/tables/list).

The following `curl` example explicitly requests a temporary credential for external data access. This request must be completed by a sufficiently privileged workspace principal.

Bash

```
curl -X POST -H "Authorization: Bearer $OAUTH_TOKEN" \  
https://<workspace-instance>/api/2.1/unity-catalog/temporary-table-credentials \  
-d '{"table_id": "<string>", "operation": "<READ|READ_WRITE>"}'
```

For details, see [POST /api/2.1/unity-catalog/temporary-table-credentials](https://docs.databricks.com/api/workspace/temporarytablecredentials/generatetemporarytablecredentials) in the Databricks REST API reference.

### Limitations

The following limitations exist:

* Not all external clients support credential vending, and support might vary depending on underlying cloud object storage.
* Delta Lake reader clients can only read tables backed by Delta Lake, and must support all enabled reader or writer protocols enabled on the table. See [Delta Lake feature compatibility and protocols](/aws/en/tables/features/feature-compatibility).
* External tables that don't use Delta Lake do not provide transactional guarantees.
* The following table types or tables with features enabled are not supported:
  + Tables with row filters or column masks.
  + Tables shared using OpenSharing.
  + Views.
  + Materialized views.
  + Lakeflow Spark Declarative Pipelines streaming tables.
  + Online tables.
  + AI Search indexes.
* Credential refresh is not supported on Iceberg 1.9.0. Use the latest Iceberg release for credential refresh.

## Path credential vending

Issued credentials allow direct access to the cloud storage location, scoped to the relevant path. They are valid for a limited time and do not grant broader access beyond the defined location or table.

### Request a temporary path credential for external data access

Support for credential vending varies by external client. Where supported, the client should automatically leverage vended credentials when a connection is configured.

This section provides an example of explicitly calling the credential vending API endpoint. Some external clients might require you to explicitly set configurations to access data and metadata in cloud object storage backing your Unity Catalog tables. You can use values returned by credential vending to configure access.

The following `curl` example explicitly requests a temporary credential for external data access. This request must be completed by a sufficiently privileged workspace principal.

Bash

```
curl -X POST -H "Authorization: Bearer $OAUTH_TOKEN" \  
https://<workspace-instance>/api/2.1/unity-catalog/temporary-path-credentials \  
-d '{"url": "<string>", "operation": "<PATH_READ|PATH_READ_WRITE|PATH_CREATE_TABLE>"}'
```

For details, see [Generate a temporary path credential](https://docs.databricks.com/api/workspace/temporarypathcredentials/generatetemporarypathcredentials) in the Databricks REST API reference.

## Volume credential vending

Public Preview

This feature is in [Public Preview](/aws/en/release-notes/release-types).

Volume credential vending lets external engines access files stored in Unity Catalog volumes with temporary, scoped credentials. The requesting principal must have `EXTERNAL USE SCHEMA` on the schema containing the volume, plus `READ VOLUME` for read access or `READ VOLUME` and `WRITE VOLUME` for write access. See [What are Unity Catalog volumes?](/aws/en/volumes/).

Unity Catalog validates permissions and returns short-lived, scoped cloud storage credentials tied to the volume's storage path. Credentials expire automatically and do not grant access beyond the specified volume.

### Requirements

* External access must be enabled on the metastore. See [Enable external data access on the metastore](/aws/en/external-access/admin#external-data-access).
* The requesting principal must have `EXTERNAL USE SCHEMA` on the schema containing the volume, plus `READ VOLUME` for read access or `READ VOLUME` and `WRITE VOLUME` for write access.
* The external engine must be able to reach the workspace URL.
* Cloud storage URLs must be accessible through firewall and network controls.

### Request a temporary volume credential for external data access

The following `curl` example explicitly requests a temporary credential for external data access.

Bash

```
curl -X POST -H "Authorization: Bearer $OAUTH_TOKEN" \  
https://<workspace-instance>/api/2.0/unity-catalog/temporary-volume-credentials \  
-d '{"volume_id": "<volume-id>", "operation": "<READ_VOLUME|WRITE_VOLUME>"}'
```

Or use the Databricks Python SDK:

Python

```
from databricks.sdk.service.catalog import VolumeOperation  
  
creds = w.temporary_volume_credentials.generate_temporary_volume_credentials(  
    volume_id=volume_id,  
    operation=VolumeOperation.READ_VOLUME,  
)
```

For details, see [POST /api/2.0/unity-catalog/temporary-volume-credentials](https://docs.databricks.com/api/workspace/temporaryvolumecredentials/generatetemporaryvolumecredentials) in the Databricks REST API reference.

### Limitations

* Managed volumes are supported for read access only. External volumes support both read and write access.
