You can query an AI Search index with the Python SDK, the REST API, or the SQL `vector_search()` function. AI Search supports ANN, hybrid, and full-text query types, along with pagination, filters, and reranking to tune retrieval quality.

For example notebooks illustrating how to create and query AI Search endpoints and indexes, see [AI Search example notebooks](/aws/en/ai-search/example-notebooks). For reference information, see the [Python SDK reference](https://api-docs.databricks.com/python/ai-search/databricks.ai_search.html).

Databricks AI Search was formerly known as Databricks Vector Search.

## How to query an AI Search index

You can only query the AI Search index using the Python SDK, the REST API, or the SQL `vector_search()` AI function.

note

If the user querying the index is not the owner of the AI Search index, the user must have the following UC privileges:

* USE CATALOG on the catalog that contains the AI Search index.
* USE SCHEMA on the schema that contains the AI Search index.
* SELECT on the AI Search index.

The default query type is `ann` (approximate nearest neighbor). For details about the different retrieval algorithms, see [Retrieval algorithms](#retrieval).

* To perform a hybrid keyword-similarity search, set the parameter `query_type` to `hybrid`. By default, hybrid search includes all text metadata columns and returns a maximum of 200 results.
* To use the reranker in a query, see [Use the reranker in a query](#rerank).

Beta

Full-text search is available as a beta feature. To perform a full-text search, set the parameter `query_type` to `FULL_TEXT`. With full-text search, you can retrieve up to 200 results based on keyword matching without using vector embeddings. Full-text queries are supported on both standard and storage-optimized endpoints. On storage-optimized endpoints, you can also create a dedicated full-text search index without embeddings. See [Create a full-text search index (Beta)](/aws/en/ai-search/create-ai-search#create-full-text-index). To search selected text columns, sort results, or return aggregate counts for full-text and hybrid queries, see [Search selected text columns, sort results, and return aggregations (Beta)](#search-sort-facet-results).

* Python SDK standard endpoint
* Python SDK storage-optimized endpoint
* REST API
* SQL

For details, see the [Python SDK reference](https://api-docs.databricks.com/python/ai-search/databricks.ai_search.html).

Python

```
# Delta Sync Index with embeddings computed by Databricks  
results = index.similarity_search(  
    query_text="Greek myths",  
    columns=["id", "field2"],  
    num_results=2  
    )  
  
# Delta Sync Index using hybrid search, with embeddings computed by Databricks  
results3 = index.similarity_search(  
    query_text="Greek myths",  
    columns=["id", "field2"],  
    num_results=2,  
    query_type="hybrid"  
    )  
  
# Delta Sync Index using full-text search (Beta)  
results4 = index.similarity_search(  
    query_text="Greek myths",  
    columns=["id", "field2"],  
    num_results=2,  
    query_type="FULL_TEXT"  
    )  
  
# Delta Sync Index with pre-calculated embeddings  
results2 = index.similarity_search(  
    query_vector=[0.9] * 1024,  
    columns=["id", "text"],  
    num_results=2  
    )
```

Python

```

index = client.get_index(index_name="vector_search_demo.vector_search.en_wiki_index")  
  
# similarity search with query vector

    query_vector=[0.2, 0.33, 0.19, 0.52],

    num_results=2  
)  
  
# similarity search with query vector and filter string  
results = index.similarity_search(  
    query_vector=[0.2, 0.33, 0.19, 0.52],  
    columns=["id", "text"],

)
```

For production applications, Databricks recommends using service principals instead of personal access tokens. In addition to improved security and access management, using service principals can improve performance by up to 100 msec per query.

The following code example illustrates how to query an index using a service principal.

sh

```
export SP_CLIENT_ID=...  
export SP_CLIENT_SECRET=...  
export INDEX_NAME=...  
export WORKSPACE_URL=https://...  
export WORKSPACE_ID=...  
  
# Set authorization details to generate OAuth token  
export AUTHORIZATION_DETAILS='{"type":"unity_catalog_permission","securable_type":"table","securable_object_name":"'"$INDEX_NAME"'","operation": "ReadVectorIndex"}'  
# If you are using an route_optimized embedding model endpoint, then you need to have additional authorization details to invoke the serving endpoint  
# export EMBEDDING_MODEL_SERVING_ENDPOINT_ID=...  
# export AUTHORIZATION_DETAILS="$AUTHORIZATION_DETAILS"',{"type":"workspace_permission","object_type":"serving-endpoints","object_path":"/serving-endpoints/'"$EMBEDDING_MODEL_SERVING_ENDPOINT_ID"'","actions": ["query_inference_endpoint"]}'  
  
# Generate OAuth token  
export TOKEN=$(curl -X POST  --url $WORKSPACE_URL/oidc/v1/token -u "$SP_CLIENT_ID:$SP_CLIENT_SECRET" --data 'grant_type=client_credentials' --data 'scope=all-apis' --data-urlencode 'authorization_details=['"$AUTHORIZATION_DETAILS"']' | jq .access_token | tr -d '"')  
  
# Get index URL  
export INDEX_URL=$(curl -X GET -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" --url $WORKSPACE_URL/api/2.0/vector-search/indexes/$INDEX_NAME | jq -r '.status.index_url' | tr -d '"')  
  
# Query AI Search index.  
curl -X GET -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" --url https://$INDEX_URL/query --data '{"num_results": 3, "query_vector": [...], "columns": [...], "debug_level": 1}'  
  
# Query AI Search index.  
curl -X GET -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" --url https://$INDEX_URL/query --data '{"num_results": 3, "query_text": "...", "columns": [...], "debug_level": 1}'
```

### Pagination

When a query requests more than 1,000 results, the results are automatically returned in pages of up to 1,000. The maximum number of results that a single query can return across all pages is 10,000. Both standard and storage-optimized endpoints support pagination.

Pagination works with all query types.

* Python SDK
* REST API

The Python SDK handles pagination transparently. When you set `num_results` to a value greater than 1,000, the SDK automatically retrieves all pages and returns the complete result set. No additional code is required.

Python

```
# The SDK automatically paginates and returns all 5000 results  
results = index.similarity_search(  
    query_text="Greek myths",  
    columns=["id", "text"],  
    num_results=5000  
)
```

When you use the REST API directly, you must handle pagination manually. If more results are available, the response includes a `next_page_token` field. To retrieve the next page of results, pass this token to the query-next-page endpoint.

See the REST API reference documentation: [POST /api/2.0/vector-search/indexes/{index\_name}/query-next-page](https://docs.databricks.com/api/workspace/vectorsearchindexes/querynextpage).

sh

```
export TOKEN=...  
export INDEX_NAME=...  
export WORKSPACE_URL=https://...  
  
# Initial query - if num_results exceeds 1000, the response includes next_page_token  
curl -X GET -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" \  
  --url $WORKSPACE_URL/api/2.0/vector-search/indexes/$INDEX_NAME/query \  
  --data '{"num_results": 5000, "query_text": "...", "columns": ["id", "text"]}'  
  
# Use next_page_token from the response to get the next page  
curl -X GET -H 'Content-Type: application/json' -H "Authorization: Bearer $TOKEN" \  
  --url $WORKSPACE_URL/api/2.0/vector-search/indexes/$INDEX_NAME/query-next-page \  
  --data '{"page_token": "<next_page_token from previous response>"}'
```

Continue calling the query-next-page endpoint with the `next_page_token` from each response until the token is empty or absent, which indicates that all results have been returned.

### Use filters on queries

A query can define filters based on any column in the Delta table. `similarity_search` returns only rows that match the specified filters.

The following table lists the supported filters.

note

For storage-optimized endpoints, results are overfetched. If you set `num_results` to `k`, more than `k` results are fetched, and the filter is applied to the fetched results. It is possible that no results will be returned even if there are results in the dataset that match the filter condition, if the score of these documents are not among the top.

| Filter operator | Behavior | Examples |
| --- | --- | --- |
| `NOT` | **Standard**: Negates the filter. The key must end with “NOT”. For example, “color NOT” with value “red” matches documents where the color is not red.  **Storage-optimized**: See [`!=` (bangeq sign) operator](/aws/en/sql/language-manual/functions/bangeqsign). | **Standard**: `{"id NOT": 2}` `{“color NOT”: “red”}`  **Storage-optimized**: `"id != 2" "color != 'red'"` |
| `<` | **Standard**: Checks if the field value is less than the filter value. The key must end with  `<`. For example, `price <` with value 200 matches documents where the price is less than 200.  **Storage-optimized**: See [`<` (lt sign) operator](/aws/en/sql/language-manual/functions/ltsign). | **Standard**: `{"id <": 200}`  **Storage-optimized**: `"id < 200"` |
| `<=` | **Standard**: Checks if the field value is less than or equal to the filter value. The key must end with  `<=`. For example, `price <=` with value 200 matches documents where the price is less than or equal to 200.  **Storage-optimized**: See [`<=` (lt eq sign) operator](/aws/en/sql/language-manual/functions/lteqsign). | **Standard**: `{"id <=": 200}`  **Storage-optimized**: `"id <= 200"` |
| `>` | **Standard**: Checks if the field value is greater than the filter value. The key must end with  `\>`. For example, `price \>` with value 200 matches documents where the price is greater than 200.  **Storage-optimized**: See [`>` (gt sign) operator](/aws/en/sql/language-manual/functions/gtsign). | **Standard**: `{"id >": 200}`  **Storage-optimized**: `"id > 200"` |
| `>=` | **Standard**: Checks if the field value is greater than or equal to the filter value. The key must end with  `\>=`. For example, `price \>=` with value 200 matches documents where the price is greater than or equal to 200.  **Storage-optimized**: See [`>=` (gt eq sign) operator](/aws/en/sql/language-manual/functions/gteqsign). | **Standard**: `{"id >=": 200}`  **Storage-optimized**: `"id >= 200"` |
| `OR` | **Standard**: Checks if the field value matches any of the filter values. The key must contain `OR` to separate multiple subkeys. For example, `color1 OR color2` with value `["red", "blue"]` matches documents where either `color1` is `red` or `color2` is `blue`.  **Storage-optimized**: See [`or` operator](/aws/en/sql/language-manual/functions/or). | **Standard**: `{"color1 OR color2": ["red", "blue"]}`  **Storage-optimized**: `"color1 = 'red' OR color2 = 'blue'"` |
| `LIKE` | **Standard**: Matches whitespace-separated tokens in a string.  **Storage-optimized**: See [`like` operator](/aws/en/sql/language-manual/functions/like). | See [Notes on usage of `LIKE`](#like). |
| No filter operator specified | **Standard**: Filter checks for an exact match. If multiple values are specified, it matches any of the values.  **Storage-optimized**: See [`=` (eq sign) operator](/aws/en/sql/language-manual/functions/eqsign) and [`in` predicate](/aws/en/sql/language-manual/functions/in). | **Standard**: `{"id": 200}` `{"id": [200, 300]}`  **Storage-optimized**: `"id = 200"` `"id IN (200, 300)"` |
