Transactions allow you to coordinate operations across multiple SQL statements and tables. All changes succeed together or roll back together, ensuring data consistency across your operations and tables. Transactions include ACID properties: atomicity, consistency, isolation, and durability. See [What are ACID guarantees on Databricks?](/aws/en/lakehouse/acid).

Transactions can be used with [stored procedures](/aws/en/sql/language-manual/sql-ref-syntax-ddl-create-procedure) and [SQL Scripting](/aws/en/sql/language-manual/sql-ref-scripting) to build mission-critical warehousing workloads.

The following example shows a transaction:

* Non-interactive
* Interactive

SQL

```
BEGIN ATOMIC  
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;  
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;  
  INSERT INTO audit_log VALUES (1, 2, 100, current_timestamp());  
END;
```

SQL

```
BEGIN TRANSACTION;  
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  
INSERT INTO audit_log VALUES (1, 2, 100, current_timestamp());  
COMMIT;
```

All three statements commit together. If any statement fails, all changes roll back and Databricks terminates the transaction without side effects.

For hands-on practice with transactions, see [Tutorial: Coordinate transactions across tables](/aws/en/transactions/tutorial).

## Requirements

To run transactions that span multiple statements or multiple tables:

* All tables written to must:
  + Be Unity Catalog managed tables (Delta Lake or Iceberg)
  + Have [Catalog commits](/aws/en/tables/features/catalog-commits) enabled
* Use supported compute:
  + For **non-interactive transactions**, use any SQL warehouse, [serverless compute](/aws/en/compute/serverless/), or [cluster](/aws/en/compute/) running Databricks Runtime 18.0 and above.
  + For **interactive transactions**, use any SQL warehouse.
  + For **transactions on OpenSharing shared assets**, use Databricks Runtime 18.1 and above.

## Transaction modes

Databricks supports two transaction modes:

| Mode | Syntax | Commit | Rollback | Best for |
| --- | --- | --- | --- | --- |
| Non-interactive | [ATOMIC compound statement](/aws/en/sql/language-manual/sql-ref-syntax-txn-begin-atomic) | Automatic on success | Automatic on error | Fixed sequences, scheduled jobs |
| Interactive | [BEGIN TRANSACTION](/aws/en/sql/language-manual/sql-ref-syntax-txn-begin); [COMMIT](/aws/en/sql/language-manual/sql-ref-syntax-txn-commit); | Manual | Manual | Conditional logic, validation and debugging, JDBC, ODBC, PyODBC |

For detailed syntax, examples, and usage patterns for both modes, see [Transaction modes](/aws/en/transactions/transaction-modes).

## Supported read sources and write sinks

Transactions allow you to read from Unity Catalog tables (Delta Lake and Iceberg), streaming tables, views, and materialized views.

Because of ACID guarantees, open table formats, such as Delta Lake and Iceberg, are supported in transactions as both read sources and write sinks. To read from non-transactional sources, use the `allow_nontransactional_read` hint. See [Read from non-transactional sources](#nontransactional-sources) and [Example: non-transactional read](#example-nontransactional-hint).

## Read from non-transactional sources

warning

Non-transactional reads are not repeatable. Concurrent changes to the source data during the transaction might result in inconsistent reads.

Transactions allow you to read from non-transactional sources. Non-transactional sources include external tables using Parquet, Avro, CSV, and JSON file formats, and [federated tables using JDBC](/aws/en/query-federation/). To read non-transactional sources, reference the source by name and use the `allow_nontransactional_read` hint.

Path-based access is not supported. If you reference a file directly by path instead, such as `` FROM parquet.`/path/to/data` ``, the transaction fails with a `PATH_BASED_ACCESS` error.

The following code example shows how to use the hint on an external table using JSON:

SQL

```
BEGIN TRANSACTION;  
-- Non-transactional source, hint required  
INSERT INTO transactional_table  
SELECT col1, col2  
FROM external_json_table  
WITH (allow_nontransactional_read = true);  
  
COMMIT;
```

### Example: non-transactional read

The following example shows a non-transactional read from an external table using Parquet. This example requires you to have an existing external location with read and write access.

See [Connect to an AWS S3 external location](/aws/en/connect/unity-catalog/cloud-storage/s3/).

To register the Parquet source as a named external table, run the following:

SQL

```
CREATE TABLE main.default.external_parquet_table  
USING PARQUET  
LOCATION 's3://my-bucket/path/to/data'; -- existing external location
```

To read both a non-transactional Parquet source and a managed table using Delta Lake in a transaction, run the following:

SQL

```
BEGIN ATOMIC  
-- Non-transactional source, hint required  
INSERT INTO transactional_table  
SELECT col1, col2  
FROM external_parquet_table  
WITH (allow_nontransactional_read = true);  
  
-- Managed table source, no hint is required  
INSERT INTO another_table  
SELECT * FROM managed_delta_table;  
END;
```

## Transaction isolation

Transactions allow repeatable reads across all statements. When you access a table in a transaction, Databricks captures a consistent snapshot of that table at first access. All subsequent reads of that table use this snapshot, so your reads remain consistent even if other users concurrently modify the same tables.

In the following example, the first query to `products` in the transaction captures a consistent snapshot:

* Non-interactive
* Interactive

SQL

```
BEGIN ATOMIC  
  SELECT * FROM products WHERE product_id = 1001;  
  SELECT * FROM products WHERE product_id = 1001;  
END;
```

SQL

```
BEGIN TRANSACTION;  
SELECT * FROM products WHERE product_id = 1001;  
SELECT * FROM products WHERE product_id = 1001;  
COMMIT;
```

Then, assume another user concurrently updates the row for `product_id = 1001` before the second query begins:

SQL

```
UPDATE products SET price = 29.99 WHERE product_id = 1001;
```

Because the snapshot was captured at first access, the second query to `products` returns the original row, not the updated one.

### Conflict detection and concurrency

Databricks uses optimistic concurrency control. Transactions proceed without locking, and conflicts are detected at commit time. When you commit, Databricks checks whether other transactions modified the same data after your transaction began. If conflicts exist, your transaction fails. For non-interactive transactions, the rollback also happens automatically. For interactive transactions, you must explicitly run `ROLLBACK` to clear the transaction state before beginning a new transaction.

Non-interactive transactions support row-level concurrency. Two transactions can modify different rows in the same data file without conflicting when [Row-level concurrency](/aws/en/optimizations/isolation/row-level-concurrency) is enabled on the target tables.

Interactive transactions support table-level concurrency.

### Conflict scenarios

| Scenario | Description |
| --- | --- |
| Write-write conflicts | Two transactions update or delete the same rows. |
| Write-read conflicts | Another transaction modified rows that your transaction read. Applies to Serializable isolation only. |
| Phantom read conflicts | Another transaction added new rows matching a predicate your transaction read. Applies to both WriteSerializable and Serializable isolation. |
| Metadata conflicts | Another transaction changed table schema or properties. |

For more details about isolation levels and conflict resolution for transactions, see [Transaction modes](/aws/en/transactions/transaction-modes). For information about isolation levels and write conflict behavior for Delta Lake tables on Databricks, see [Optimization recommendations on Databricks](/aws/en/optimizations/).

## How transactions appear in the Delta log

Each successful transaction appears as a single entry in the table's Delta log, regardless of how many individual statements ran within the transaction. This allows for a clean audit trail and simplifies rollback operations.

Individual operations within a transaction are available as JSON metadata in the Delta log entry for the transaction.

## Error handling and rollback

The following table describes how error rollbacks occur for both transaction types:

| Scenario | Behavior for non-interactive transactions | Behavior for interactive transactions |
| --- | --- | --- |
| Statement failure | Any statement that raises an error causes immediate automatic rollback. | You must explicitly run [ROLLBACK](/aws/en/sql/language-manual/sql-ref-syntax-txn-rollback) to discard changes if the session is still active. |
| Failed validation logic or business rules | Use [`SIGNAL`](/aws/en/sql/language-manual/control-flow/signal-stmt) to throw an exception and trigger automatic rollback. | Run [ROLLBACK](/aws/en/sql/language-manual/sql-ref-syntax-txn-rollback) to discard changes. |
| Session disconnect | The transaction automatically rolls back. | The transaction automatically rolls back. |
| Timeout | Automatically rolls back after 48 hours total duration. | Automatically rolls back after 10 minutes of inactivity or 48 hours total duration (see [Limitations](#limitations)). The transaction is terminated without side effects, but you must explicitly run [ROLLBACK](/aws/en/sql/language-manual/sql-ref-syntax-txn-rollback) to clear the transaction state if the session is still active. |

For interactive transactions, you can explicitly roll back using the [ROLLBACK](/aws/en/sql/language-manual/sql-ref-syntax-txn-rollback) statement. This allows you to discard changes based on validation logic or business rules, or after a statement failure when the session remains active.

## Best practices

Follow these practices to reduce conflicts and optimize transaction performance.

### Avoid conflicts

* **Keep transactions short**: Long-running transactions increase conflict likelihood and hold resources longer.
* **Validate early**: Check preconditions at the beginning of a transaction to fail fast.
* **Databricks recommends non-interactive transactions for most use cases**: Non-interactive transactions use row-level concurrency. See [Non-interactive transactions](/aws/en/transactions/transaction-modes#non-interactive).
* **Retry on conflicts**: When conflicts occur, retry the transaction with fresh data.

## Use transactions from different clients

Transactions work across various client interfaces:

* **SQL Editor and notebooks**: Use [`BEGIN ATOMIC ... END;`](/aws/en/sql/language-manual/sql-ref-syntax-txn-begin-atomic) or [`BEGIN TRANSACTION; ... COMMIT;`](/aws/en/sql/language-manual/sql-ref-syntax-txn-begin) syntax directly in SQL cells or use `spark.sql()` in Python/Scala notebooks. See [Transaction modes](/aws/en/transactions/transaction-modes).
* **JDBC applications**: Use JDBC API methods (`setAutoCommit(false)`, `commit()`, `rollback()`) with the [Databricks JDBC driver](https://github.com/databricks/databricks-jdbc/releases) version 3.0.5 and above. See [Example: Use transactions](/aws/en/integrations/jdbc-oss/example#transactions). For a list of unsupported JDBC operations within transactions, see [Unsupported JDBC operations](/aws/en/transactions/transaction-modes#int-jdbc-not-supported).
* **ODBC applications**: Use the [Databricks ODBC Driver](/aws/en/integrations/odbc/) version 2.10.0 and above. For a list of unsupported ODBC operations within transactions, see [Unsupported ODBC operations](/aws/en/transactions/transaction-modes#int-odbc-not-supported).
* **Python applications**: Use the Databricks SQL Connector with `autocommit=False`. See [Databricks SQL Connector for Python](/aws/en/dev-tools/python-sql-connector). For a list of unsupported Python connector operations within transactions, see [Unsupported Python connector operations](/aws/en/transactions/transaction-modes#int-python-not-supported).
* **Statement Execution API**: Run transactions using SQL syntax through API calls. See [Use with Statement Execution API](/aws/en/transactions/transaction-modes#ni-api).
