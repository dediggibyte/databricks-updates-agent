Declarative Automation Bundles (formerly known as Databricks Asset Bundles) are created, deployed to a workspace, then managed programmatically from your local development environment or directly in the workspace UI. Collaborating on bundles directly in the workspace allows for more rapid iteration and testing before moving to production.

For more information about bundles, see [What are Declarative Automation Bundles?](/aws/en/dev-tools/bundles/).

tip

If you don't want to use the workspace UI to create and manage your bundles, but would still like to develop bundles in the workspace, you can use the Databricks CLI from the workspace web terminal, just as you would from your local terminal. See [Run Databricks CLI commands](/aws/en/compute/web-terminal#cli-workspace).

## What are the installation requirements of bundles in the workspace?

You do not have to install anything locally to use bundles in the workspace, but there are Databricks workspace requirements:

* Workspace files must be enabled. See [What are workspace files?](/aws/en/files/workspace).
* You must have a Git folder in which to create the bundle. To create a Git folder, see [Clone a repo](/aws/en/repos/git-operations-with-repos#clone-repo).
* Serverless compute must be enabled. See [Serverless compute requirements](/aws/en/compute/serverless/#requirements).
* Bundles in the workspace are compatible with [serverless egress control](/aws/en/security/network/serverless-network-security/network-policies). However, if you need to build Python projects using packages from the public PyPi registry, use a private mirror instead.

The version of the Databricks CLI used to deploy a bundle and run resources in the workspace is provided in the **Deploy** dialog.

## Do I have to understand how to author YAML to use bundles in the workspace?

No. Declarative Automation Bundles in the workspace provides a UI for managing bundles so that you can collaborate on bundles that were developed by other users in their local development environments. You do not need to learn YAML or know how to use the Databricks CLI to work with bundles in the workspace.

## I already manage my workflows in the Databricks UI. Why should I use bundles?

Declarative Automation Bundles allows you to source control all of the files needed for your workflows. Bundles in the workspace provides a mechanism for workspace users to edit, commit, test, and deploy updates through the UI. Users launch a bundle project from within a Git folder.

## I have a bundle in a GitHub repository. How can I edit it in the Databricks workspace?

A folder is identified as a bundle by Databricks if a `databricks.yml` file exists at the root of the folder. Create a Git folder in Databricks for your GitHub repository, and Databricks will recognize it as a bundle.

## Is all bundle configuration supported in the workspace?

Almost all of your existing bundles can be managed in the workspace. However, [Python for Declarative Automation Bundles](/aws/en/dev-tools/bundles/python/) is not currently supported in the workspace.

## How do I share a bundle for collaboration?

Bundles inherit the permissions of their parent Git folder, so to share a bundle, share the Git folder that contains your bundle project with your collaborators. See [Collaborate, review, and deploy to production](/aws/en/dev-tools/bundles/workspace-deploy#share).

## How can I move a bundle to production in the workspace?

Bundles define a complete project to deploy, the infrastructure necessary to run the project, and the deployment targets. These targets are typically your development and production modes. Declarative Automation Bundles in the workspace makes it easy to switch targets, deploy, and run workflows from the UI. See [Deploy bundles and run workflows from the workspace](/aws/en/dev-tools/bundles/workspace-deploy).

## Can I deploy my bundle to another Databricks workspace from the bundle editor?

No, this is not supported. Databricks recommends developing in the bundle editor in the workspace, committing changes by syncing the Git folder, then setting up a CI/CD workflow (for example, a GitHub Action) that uses the CLI to deploy the bundle to another workspace.
